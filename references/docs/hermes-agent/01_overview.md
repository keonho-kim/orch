# Hermes Agent 개요

이 문서는 `hermes-agent`가 어떤 종류의 시스템인지, 어떤 운영 모델을 전제하는지, 그리고 코드베이스가 어떤 우선순위를 보여 주는지 정리한다. 판단의 중심 근거는 `run_agent.py`, `gateway/run.py`, `hermes_state.py`, `hermes_cli/`, `tools/`, `pyproject.toml`, `Dockerfile`, `.github/workflows/`, `tests/`다. 결론부터 말하면 Hermes는 “대화형 AI”보다 “여러 표면에서 계속 살아 있는 `agent runtime(에이전트 실행 환경)`”에 더 가깝다. 핵심 문제도 답변 품질 자체보다 세션 지속성, 권한 경계, 도구 실행, 상태 저장, 배포 유연성 쪽에 놓여 있다.

짧게 말해 Hermes는 하나의 챗봇이 아니다. 파일을 읽고, 셸 명령을 실행하고, 브라우저를 조작하고, 여러 메시징 채널과 IDE에서 같은 세션을 이어 쓰고, 예약 작업과 장기 기억을 관리하는 운영 중심 소프트웨어다. 비전문가 입장에서 보면 “질문에 답하는 모델”이 아니라 “사용자가 직접 운영하는 디지털 작업자”에 가깝다.

## 이 문서가 보는 관점

여기서 중요한 것은 저장소를 제품 소개가 아니라 `reverse engineering(역분석)` 대상으로 읽는다는 점이다. 그래서 다음 구분을 유지한다.

- 직접 확인한 사실: 소스 코드, 설정 파일, CI 설정, 테스트 구조, 패키징 경로
- 강한 추론: 제품 철학, 운영자 가정, 우선순위
- 제한적 외부 보강: `MCP(모델 컨텍스트 프로토콜)`, `ACP(에이전트 클라이언트 프로토콜)` 같은 외부 프로토콜의 일반적 의미

이 구분이 중요한 이유는 Hermes가 표면적으로는 범용 `AI assistant(AI 비서)`처럼 보이지만, 실제 구현은 훨씬 더 많은 운영 문제를 품고 있기 때문이다.

## Hermes는 어떤 종류의 시스템인가

코드 구조상 Hermes는 최소한 다섯 가지 제품 성격을 동시에 가진다.

- 대화형 `CLI(명령행 인터페이스)`: `hermes_cli/main.py`, `cli.py`
- 장기 실행 메시징 `gateway(게이트웨이)`: `gateway/run.py`, `gateway/platforms/*`
- 편집기 연동용 `ACP server(ACP 서버)`: `acp_adapter/server.py`
- `OpenAI-compatible API(오픈AI 호환 API)`: `gateway/platforms/api_server.py`
- 외부 도구 노출용 `MCP server(MCP 서버)`: `mcp_serve.py`

여기에 `cron(크론)` 자동화, 웹 대시보드, 원격 실행 백엔드, 기억 제공자 플러그인, 브라우저 자동화, 음성 입출력까지 붙어 있다. 따라서 Hermes를 단순 `code agent(코드 에이전트)`로 부르면 좁고, `workflow engine(워크플로 엔진)`로 부르면 반대로 핵심을 놓친다. 현재 저장소 구성만 놓고 보면 가장 정확한 표현은 “장기 세션과 도구 실행을 중심에 둔 다표면 `agent platform(에이전트 플랫폼)`”이다.

## 제품 포지션: 로컬 우선보다 운영자 우선

Hermes는 흔히 `local-first(로컬 우선)` 범주로 읽힐 수 있지만, 코드가 드러내는 실제 태도는 `operator-first(운영자 우선)`에 더 가깝다. 상태 루트는 기본적으로 `HERMES_HOME` 아래에 놓이지만, 실행 위치는 로컬에 고정되지 않는다. `tools/environments/*`는 local, Docker, SSH, Modal, Daytona, Singularity를 함께 다루고, 게이트웨이와 자동화는 장기 실행 프로세스를 전제로 한다.

이 구조는 몇 가지 운영 가정을 뜻한다.

- 사용자는 중앙 SaaS `control plane(제어 평면)`보다 자기 소유 환경을 선호한다.
- 제어 프로세스와 실제 부작용 실행 위치는 분리될 수 있다.
- 설치, 인증, 운영 책임은 사용자 또는 운영 팀이 감수한다.

즉 Hermes는 “바로 켜서 쓰는 소비자 앱”보다 “설정과 운영을 감당할 수 있는 개발자용 런타임”에 가깝다.

## 코드가 드러내는 개발 철학

### 1. 표면이 늘어나도 코어는 하나로 유지한다

`run_agent.py`의 `AIAgent(AI 에이전트)`가 CLI, 게이트웨이, `ACP`, API 서버, `cron`의 공통 코어 역할을 맡는다. 이는 기능 중복을 줄이고 행동 일관성을 높이는 대신, 중심 파일이 거대해지는 비용을 감수하는 선택이다. 실제로 `run_agent.py`, `gateway/run.py`, `cli.py`는 모두 대형 파일이며, 정책과 예외 처리가 이 층에 강하게 집중되어 있다.

이 선택은 현실적이다. 다표면 제품에서는 각 표면을 별도 구현으로 분리할수록 기능 차이와 회귀가 빨리 커진다. Hermes는 그 위험보다 중앙 응집의 비용을 택했다.

### 2. `tool(도구)`보다 `skill(스킬)`이 먼저 온다

`tools/skills_tool.py`, `tools/skill_manager_tool.py`, `agent/prompt_builder.py`, `skills/`, `optional-skills/`를 보면 Hermes는 많은 능력을 코드 하드포크 대신 스킬과 프롬프트 계층으로 확장하려 한다. 바이너리 처리, 인증, 스트리밍, 강한 계약이 필요한 영역만 진짜 도구로 승격하고, 나머지는 문서 기반 절차와 조합으로 푸는 쪽에 가깝다.

이 방식은 유지보수에 유리하다. 모든 기능을 Python 코드에 박아 넣지 않아도 되고, 운영자가 제품을 스스로 늘릴 수 있다. 반대로 품질 편차와 `prompt semantics(프롬프트 구조)` 의존성이 커지는 단점도 따른다.

### 3. 모델 제공자는 언제든 바뀐다고 가정한다

`hermes_cli/runtime_provider.py`, `hermes_cli/main.py`, `pyproject.toml`을 보면 Hermes는 특정 `provider(모델 제공자)`에 아키텍처를 잠그지 않는다. OpenAI, Anthropic, OpenRouter, Copilot, Gemini 계열, 커스텀 엔드포인트까지 흡수하며, 내부에서는 `chat_completions`, `codex_responses`, `anthropic_messages` 같은 서로 다른 API 형식을 공통 루프로 맞춘다.

이는 “좋은 에이전트는 좋은 모델 하나에 달려 있다”가 아니라 “모델 시장의 변동성 자체가 제품 전제”라는 판단을 반영한다. 이 선택은 내구성이 있지만, 런타임 해석 계층을 두껍게 만든다.

### 4. 안전은 능력 제거보다 경계 분리로 푼다

`tools/approval.py`, `tools/path_security.py`, `tools/terminal_tool.py`, `tools/mcp_tool.py`, `hermes_cli/web_server.py`, `Dockerfile`, `hermes_logging.py`, `supply-chain-audit.yml`을 함께 보면 Hermes의 안전 모델은 봉쇄형보다 다층 경계형이다.

- 위험 명령은 승인 흐름을 탄다.
- 파일 경로는 허용 범위를 벗어나지 않도록 검사한다.
- 원격 실행 백엔드는 호스트와 부작용을 분리한다.
- `MCP` 하위 프로세스에는 제한된 환경 변수만 전달한다.
- 대시보드는 로컬 CORS와 세션 토큰으로 민감 API를 제한한다.
- 로그는 회전 저장과 비밀값 `redaction(민감정보 마스킹)`을 기본 전제로 둔다.

즉 Hermes는 강한 도구 능력을 유지하되, 권한과 실행 위치를 분리하는 식으로 사고 반경을 줄이려 한다.

## Hermes가 줄이려는 운영자 고통

코드 기준으로 보면 Hermes가 해결하려는 문제는 꽤 명확하다.

### 채널이 바뀌면 작업이 끊기는 문제

CLI, 메시징 게이트웨이, `ACP`, API 서버가 같은 코어를 공유하기 때문에 사용자는 입력 채널을 바꿔도 세션 운영 방식 자체를 바꾸지 않아도 된다. 이는 일반 챗봇보다 “어디서든 같은 실행 주체를 이어 쓰기”에 가깝다.

### 긴 세션이 점점 무거워지는 문제

`agent/context_compressor.py`, `agent/context_engine.py`, `prompt_caching` 관련 경로, 게이트웨이의 자동 압축 로직은 모두 장기 세션 비용을 줄이기 위한 장치다. Hermes는 짧은 질의응답보다 장기 작업 세션을 더 중요한 사용 시나리오로 본다.

### 예전 대화와 누적 지식을 다시 못 쓰는 문제

`hermes_state.py`의 SQLite 저장소, `FTS5(전문 검색)`, `tools/session_search_tool.py`, 메모리 관리자와 메모리 플러그인 구조는 “지난 대화와 장기 상태를 다시 불러오는 능력”을 코어 자산으로 취급한다.

### 한 가지 실행 환경에 묶이는 문제

`tools/terminal_tool.py`와 `tools/environments/*`는 실행 위치를 추상화한다. 에이전트는 한 대의 로컬 노트북 안에만 머무를 필요가 없고, 제어와 실행을 분리한 `hybrid runtime(혼합 실행 환경)`가 가능하다.

## 어떤 사용자를 겨냥하는가

Hermes의 1차 사용자상은 비교적 분명하다.

- 셀프호스팅이나 자가 운영을 감당할 수 있는 개발자
- 파일 수정, 셸 실행, 세션 검색, 예약 실행이 필요한 고급 사용자
- 하나의 에이전트를 여러 채널과 워크스페이스에서 이어 쓰고 싶은 개인 운영자
- `skill`, `plugin`, `MCP`, 원격 실행 백엔드로 제품을 확장하려는 팀

반대로 다음 유형에는 덜 맞는다.

- 완전 관리형 경험을 기대하는 일반 소비자
- 강한 폐쇄형 `sandbox(격리 실행 환경)`를 기본값으로 요구하는 조직
- 웹 채팅 UI 하나만으로 충분한 사용 시나리오
- API 키, 토큰, 프로필, 백그라운드 프로세스 관리를 손대고 싶지 않은 사용자

`web/src/App.tsx`와 `hermes_cli/web_server.py`가 보여 주는 대시보드도 채팅 UI보다 운영 콘솔에 가깝다. 상태, 세션, 로그, 분석, `cron`, 스킬, 설정, 환경 변수 관리가 한 화면 체계 안에 함께 놓인다.

## 언어와 프레임워크 선택이 시사하는 점

Hermes의 중심 런타임은 Python 3.11+다. `pyproject.toml`과 엔트리포인트 구성을 보면 Python이 코어 통합 언어로 선택되었고, 그 위에 Node 생태계가 보조적으로 붙는다.

- Python: 에이전트 코어, 도구, 게이트웨이, 저장소, 패키징, 테스트
- Node: 브라우저 자동화 의존성, 웹 대시보드 번들, WhatsApp 브리지
- React + Vite: 운영 대시보드
- FastAPI/Uvicorn: 웹 대시보드 백엔드와 일부 HTTP 표면

이 조합은 프런트엔드 중심 제품보다 런타임 통합 중심 제품의 전형에 가깝다. 팀의 우선순위가 “예쁜 UI”보다 “다양한 실행 환경과 도구를 빨리 통합하는 것”에 있음을 보여 준다.

## 핵심 의존성과 그 의미

중요한 의존성은 아키텍처 방향을 직접 드러낸다.

| 의존성 계열 | 대표 패키지 | 아키텍처적 의미 |
|---|---|---|
| 모델 SDK | `openai`, `anthropic` | 복수 응답 형식을 공통 루프로 흡수 |
| 통신/복구 | `httpx`, `requests`, `tenacity` | 외부 API 실패를 정상 상태로 가정 |
| 터미널 UX | `prompt_toolkit`, `rich` | 단순 REPL이 아니라 장기 상호작용형 CLI 지향 |
| 웹/API | `fastapi`, `uvicorn` | 대시보드와 HTTP 인터페이스를 공용 운영 인터페이스로 취급 |
| 메시징 | `python-telegram-bot`, `discord.py`, `slack-bolt` | 게이트웨이를 제품 핵심으로 간주 |
| 브라우저 자동화 | `agent-browser`, `@askjo/camofox-browser` | 텍스트 중심을 넘는 실행 능력 확보 |
| 프로토콜 | `mcp`, `agent-client-protocol` | 외부 에이전트 생태계와 상호운용 가능한 허브 전략 |
| 자동화 | `croniter` | 예약 실행을 부수 기능이 아니라 기본 능력으로 간주 |
| 메모리 | `honcho-ai` 및 메모리 플러그인 계열 | 세션 외부 장기 상태를 교체 가능 계층으로 분리 |

특히 `MCP`와 `ACP`를 함께 다루는 점이 중요하다. 하나는 외부 도구를 Hermes 안으로 들여오는 경로이고, 다른 하나는 Hermes 자체를 편집기 `client(클라이언트)`와 연결하는 경로다. 즉 Hermes는 “도구를 소비하는 시스템”이면서 동시에 “다른 클라이언트가 붙을 수 있는 실행 주체”이기도 하다.

## 고수준 `trust model(신뢰 모델)`

Hermes는 누구를 믿고 누구를 의심하는지가 비교적 분명하다.

- 운영자는 기본적으로 신뢰한다.
- 모델은 강력하지만 신뢰할 수 없는 제안자로 취급한다.
- 외부 사용자 입력과 첨부물은 불신한다.
- 작업 공간 지침 파일과 스킬도 `prompt injection(프롬프트 인젝션)` 표면으로 본다.
- 원격 도구 서버와 하위 프로세스는 환경 변수와 접두어로 제한한다.
- 위험 명령은 즉시 실행보다 승인과 격리를 우선한다.

이 모델은 고정형 기업용 정책 엔진과는 다르다. Hermes는 높은 실행력을 유지하면서도, 승인 흐름과 실행 위치 분리로 위험을 줄이는 현실적 절충을 택한다.

## 비슷한 계열과 비교해 보면

Hermes를 더 선명하게 이해하려면 비슷한 계열의 도구들과 어떤 점이 다른지 봐야 한다.

### `chat assistant(채팅 비서)`와의 차이

채팅 비서는 대개 한 번의 질의응답과 UI 경험이 중심이다. Hermes는 세션 저장, 로그, 검색, 예약 실행, 백그라운드 프로세스, 메시징 허브를 더 중요하게 본다. 따라서 질문 응답기보다 장기 작업기 쪽에 가깝다.

### `workflow engine(워크플로 엔진)`와의 차이

워크플로 엔진은 정형화된 단계와 재현 가능한 DAG를 선호한다. Hermes는 루프 내부에서 모델이 도구를 선택하고 예외를 복구하는 비정형 오케스트레이션에 더 가깝다. 그래서 유연성은 높지만, 완전한 결정성과 감사 용이성은 낮다.

### `code agent(코드 에이전트)`와의 차이

코드 수정과 셸 실행은 Hermes의 중요한 능력이지만 전부는 아니다. 메시징 플랫폼, 브라우저, 음성, 홈 자동화, `MCP`, 장기 기억까지 포함하면 Hermes는 특정 개발 워크플로에만 최적화된 도구라기보다 범용 작업 런타임에 가깝다.

### `local-first agent(로컬 우선 에이전트)`와의 차이

상태는 로컬 또는 자기 소유 경로에 두지만, 실행은 원격 백엔드로 보낼 수 있다. 따라서 순수 로컬 우선보다는 “로컬 상태 + 선택적 원격 실행”이라는 혼합형이 더 정확하다.

## 가장 강한 장점

- 다표면 제품을 하나의 코어 루프로 묶어 기능 일관성을 확보한다.
- 세션 저장, 검색, 압축, 계보 추적이 실제 코드와 저장 구조로 구현되어 있다.
- 도구 레지스트리, `toolset(도구 묶음)`, 플러그인, `MCP`를 조합해 확장 경계를 비교적 선명하게 관리한다.
- 승인, 경로 보안, 원격 백엔드, 대시보드 인증, 로그 마스킹 등 실전 운영용 안전 장치가 넓게 깔려 있다.
- 테스트 수가 많고, Docker/Nix/패키징/CI가 함께 있어 단순 실험 저장소보다 제품화 수준이 높다.

## 가장 눈에 띄는 한계

- 중앙 코어 파일이 매우 커서 변경 반경이 넓다.
- 제공자, 플랫폼, 백엔드 조합이 많아질수록 회귀 위험이 빠르게 커진다.
- 상태가 SQLite, 파일, 프로필별 경로, 백엔드별 저장소에 분산돼 디버깅 난도가 높다.
- 운영자 친화적 설계인 만큼 설치와 인증, 프로세스 관리 부담이 사용자 측에 남는다.
- 강한 실행력을 유지하는 대신, 보안 정책이 단일 엔진이 아니라 여러 계층에 흩어져 있다.

## 1차 차별점

Hermes를 일반적인 오픈소스 에이전트와 구분하는 가장 신뢰할 만한 차별점은 다음 셋이다.

- 메시징 게이트웨이, IDE 프로토콜, HTTP API, `MCP` 서버를 하나의 코어 위에 동시에 올린 다표면 구조
- 세션 DB, 압축, 세션 검색, 기억 플러그인까지 포함한 장기 세션 운영 감각
- 로컬 실행, 컨테이너, SSH, 클라우드 백엔드를 하나의 터미널 추상화로 묶은 실행 위치 분리

많은 저장소가 “도구를 부르는 모델” 수준에서 멈추는 데 반해, Hermes는 “계속 살아 있는 운영 주체”를 만들려는 의도가 훨씬 강하다.

## 성숙도와 아직 남은 거친 면

코드상 성숙도의 신호는 분명하다.

- `tests/` 아래 569개 파일 규모의 넓은 회귀 범위
- `tests`, `docker-publish`, `nix`, `supply-chain-audit`를 포함한 CI 체계
- Docker 이미지, Homebrew 패키징, `setup-hermes.sh`, `uv.lock` 기반 설치 경로
- 로그 회전, 세션 태그, 대시보드 상태/로그/분석 API

반면 아직 거친 면도 분명하다.

- 기능이 커질수록 중심 파일이 더 커지는 경향
- 동기 코어와 비동기 표면을 스레드/브리지로 잇는 구조적 복잡성
- 넓은 옵션 집합 때문에 설정 해석 경로가 복잡해진 점

즉 Hermes는 프로토타입 단계를 이미 넘었지만, 완전히 정제된 플랫폼 프레임워크라고 보기에는 아직 코어 집중에서 오는 비용이 크다.

## 요약

Hermes의 실제 정체성은 다표면 `agent runtime(에이전트 실행 환경)`이자 운영자 중심 `agent platform(에이전트 플랫폼)`이다. 핵심 가치는 멋진 답변 하나보다 장기 세션 유지, 도구 실행, 상태 저장, 채널 연속성, 실행 위치 분리에 있다. 강점은 공통 코어 재사용, 장기 세션 운영, 넓은 확장 경계, 현실적인 안전 장치에 있고, 한계는 거대한 중심 파일, 넓은 통합 범위, 분산 상태 관리, 운영 부담에 있다. Hermes를 이해할 때는 “똑똑한 챗봇”보다 “사용자가 직접 운영하는 개인용 에이전트 인프라”로 보는 편이 정확하다.

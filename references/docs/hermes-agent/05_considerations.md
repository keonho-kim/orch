# Hermes Agent 설계 고려사항

이 문서는 Hermes를 참고해 비슷한 시스템을 만들려는 팀을 위한 엔지니어링 판단 정리다. 결론부터 말하면 Hermes에서 가장 직접적으로 배울 만한 것은 “공통 에이전트 코어를 여러 표면에 재사용하는 구조”, “장기 세션을 저장과 검색 가능한 운영 자산으로 다루는 방식”, “실행력은 유지하되 승인·격리·프로필 분리로 위험을 줄이는 태도”다. 반대로 그대로 베끼기 가장 위험한 부분은 초대형 중앙 오케스트레이터, 지나치게 넓은 플랫폼/백엔드 범위, 그리고 파일·SQLite·플러그인·원격 실행이 얽힌 운영 복잡도다.

따라서 이 문서의 핵심 판단은 단순하다. Hermes는 그대로 베낄 대상이 아니라, 계층별 차용 우선순위를 정해 가져와야 할 참고 구조다. 새 팀이라면 모델 호출 품질보다 세션 저장, 권한 경계, 실행 백엔드, 인터럽트 처리, 관측성을 먼저 설계하는 편이 훨씬 낫다.

## 먼저 그대로 가져갈 만한 것

### 공통 코어 재사용 전략

Hermes가 가장 잘한 선택은 CLI, 게이트웨이, `ACP`, API 서버, `cron`을 따로 구현하지 않고 하나의 `AIAgent` 코어 위에 얹었다는 점이다. 이 패턴은 여러 표면에서 행동 일관성을 확보하는 데 매우 강하다.

얻는 이점은 분명하다.

- 기능 차이를 줄이기 쉽다.
- 테스트를 재사용할 수 있다.
- 새 표면을 붙일 때 코어를 다시 만들 필요가 없다.

다만 이 전략을 쓰려면 인터페이스 계층이 코어 정책을 침범하지 않도록 엄격한 래퍼 경계를 유지해야 한다. Hermes도 이 부분을 완전히 정제했다고 보기는 어렵지만, 방향 자체는 매우 유효하다.

### 자체 등록형 `tool registry(도구 레지스트리)`

`tools/registry.py`와 `model_tools.py`가 보여 주는 자체 등록형 도구 구조는 실무 가치가 높다. 중앙 스위치문 없이 도구가 등록되고, `toolset`, 플러그인 도구, `MCP` 도구까지 같은 경로로 흡수된다.

여러 팀원이 도구를 병렬로 늘리는 환경에서는 이 패턴이 특히 중요하다. 코드 변경 지점이 분산되더라도 런타임 노출 경로는 하나로 유지되기 때문이다.

### 세션 저장소를 로그가 아니라 핵심 저장 체계로 보는 태도

`hermes_state.py`의 SQLite `SessionDB`는 Hermes에서 가장 안정적인 축 중 하나다. 세션 저장, 메시지 저장, 전문 검색, 제목, 계보, 비용 통계가 함께 들어 있다.

장기 실행형 에이전트에서 세션 저장은 부가 기능이 아니라 운영의 중심이다. 단순 JSON 로그만으로는 검색, 복구, 요약, 계보 추적이 곧 한계에 부딪힌다. 이 점은 다른 팀도 거의 그대로 가져갈 가치가 있다.

### `profile(프로필)` 기반 격리

`hermes_constants.py`, `hermes_cli/profiles.py`, `hermes_cli/main.py`가 보여 주듯 Hermes는 프로필을 단순 UI 기능이 아니라 저장소와 실행의 기본 단위로 다룬다.

- 각 프로필이 별도 `HERMES_HOME`을 가진다.
- 설정, 키, 세션, 스킬, 기억, 로그가 분리된다.
- per-profile HOME을 통해 하위 프로세스의 git/ssh/gh 설정까지 분리할 수 있다.

멀티 인스턴스 요구가 조금이라도 예상된다면, 이 분리를 초기에 넣는 편이 훨씬 싸다.

### 실전 운영용 안전 장치

Hermes는 실행력을 포기하지 않으면서도 안전 장치를 꽤 깊게 넣었다.

- 위험 명령 승인
- 경로 이탈 방지
- 격리 실행 백엔드
- 대시보드 세션 토큰
- 로그 마스킹
- 공급망 감사 워크플로

이 조합은 그대로 복사하기보다 원리를 차용할 가치가 높다. “강한 능력 + 다층 경계”라는 방향 자체가 장기적으로 유효하다.

### 운영 콘솔과 로그 계층

웹 대시보드, `hermes logs`, 세션 태깅 로그, 런타임 상태 API는 Hermes가 에이전트를 “운영 대상”으로 본다는 사실을 잘 보여 준다. 비슷한 시스템을 만드는 팀이라면 채팅 UI보다 먼저 로그와 상태 조회 구조를 설계하는 편이 낫다.

## 조정해서 가져와야 할 것

### 다중 `provider(모델 제공자)` 런타임

Hermes의 제공자 해석 계층은 강력하지만, 거의 별도 모듈처럼 느껴질 정도로 복잡하다. 여러 인증 방식, 여러 API 모드, 커스텀 엔드포인트, 자격증명 풀, 대체 제공자 체인이 얽혀 있다.

이 구조는 공급자 교체가 잦은 제품에는 분명 이점이 있다. 하지만 모든 팀이 처음부터 이 수준의 유연성을 필요로 하지는 않는다. 작은 팀이라면 다음 순서가 더 건강하다.

1. 기본 제공자 1개
2. 대체 제공자 1개
3. 공통 내부 메시지 포맷
4. 그다음에야 커스텀 엔드포인트와 풀, 자동 감지 추가

Hermes의 설계를 베끼더라도 단계적으로 가져와야 한다.

### 메시징 `gateway(게이트웨이)` 중심 구조

Hermes는 게이트웨이를 부수 기능이 아니라 제품 허브로 본다. 이 판단은 다표면 제품에는 맞지만, 초기 제품에는 과한 투자일 수 있다.

게이트웨이를 빨리 붙이면 좋은 점:

- 장기 실행과 원격 사용 시나리오를 바로 검증할 수 있다.
- 플랫폼 간 세션 연속성을 실험하기 쉽다.

게이트웨이를 너무 빨리 붙일 때 나쁜 점:

- 권한, 인터럽트, 승인, 스트리밍, 큐잉, 배달 실패를 동시에 떠안게 된다.
- UI보다 운영 문제가 먼저 터진다.

즉 게이트웨이 허브 구조는 좋은 목표지만, 작은 팀은 CLI와 하나의 외부 채널 정도로 시작하는 편이 보통 더 낫다.

### 여러 실행 백엔드

local, Docker, SSH, Modal, Daytona, Singularity를 동시에 지원하는 것은 인상적이지만, 실무적으로는 “어디서나 실행”이 아니라 “어디서나 다른 종류의 실패를 겪는다”는 뜻이기도 하다.

차용 시 조정 원칙은 단순하다.

- 제품 가치가 보안/격리에 있으면 local + container 두 개로 시작한다.
- 원격 지속성이 중요하면 local + ssh 또는 local + daytona/modal 조합이 낫다.
- HPC 수요가 실제로 있기 전에는 Singularity는 미루는 편이 낫다.

Hermes의 넓은 범위는 성숙한 운영자가 있을 때 의미가 크고, 그렇지 않으면 테스트 매트릭스만 폭증시킨다.

### `plugin(플러그인)` 자유도

Hermes는 일반 플러그인, 기억 제공자, 문맥 엔진, `MCP` 서버를 모두 지원한다. 확장성은 좋지만 품질 편차와 호환성 부담도 같이 온다.

이 구조를 차용한다면 다음 원칙이 안전하다.

- 일반 플러그인과 훅은 비교적 자유롭게 열 수 있다.
- 기억과 문맥처럼 핵심 동작에 직접 닿는 부분은 단일 선택으로 제한한다.
- 프로젝트 로컬 플러그인은 명시적 opt-in으로 둔다.

Hermes가 바로 이 방식을 택한 이유를 가볍게 보면 안 된다. 모든 것을 동시에 합성 가능하게 만들면 결국 코어 동작의 기준이 흐트러진다.

## 다시 설계하는 편이 나은 것

### 초대형 중앙 파일

`run_agent.py`, `gateway/run.py`, `cli.py`는 Hermes의 가장 큰 생산성과 가장 큰 부채를 동시에 대표한다. 기능을 빨리 붙일 때는 효율적이지만, 시간이 지나면 탐색성, 변경 안정성, 코드 소유권 분리가 모두 어려워진다.

새 팀이라면 최소한 다음 축으로 미리 쪼개는 편이 낫다.

- 턴 오케스트레이션
- 모델 호출과 응답 파싱
- 도구 실행
- 압축/문맥 엔진
- 세션 저장
- 표면별 콜백과 렌더링
- 로그와 상태 갱신

Hermes는 이미 충분히 커져서 중앙 파일 비용이 눈에 띄는 단계에 들어가 있다.

### 설정 해석의 분산

`config.yaml`, `.env`, 프로필, 런타임 override, 표면별 추가 규칙이 함께 작동하는 구조는 유연하지만 추적하기 어렵다. 특히 제공자 해석과 표면별 설정 차이는 재현성을 해칠 수 있다.

새 시스템이라면 초기에 다음을 더 엄격히 하는 편이 좋다.

- 설정 로드 계층을 하나로 통합한다.
- 표면별 override도 명시적 구조로 둔다.
- “어디서 이 값이 왔는지”를 항상 디버그 가능하게 만든다.

### 상태 저장의 다중화

Hermes의 파일 + SQLite 혼합 전략은 실용적이지만 정합성 경계가 늘어난다. 세션 데이터, 게이트웨이 라우팅 상태, 기억 파일, 스킬 파일, `cron/jobs.json`, 로그, 런타임 상태가 따로 존재한다.

감사와 운영 진단에는 유리하지만, 단순성을 원하는 팀이라면 과하다. 처음부터 이렇게 넓게 가져갈 필요는 없다.

### 지나치게 넓은 조합 폭

메시징 플랫폼, 실행 백엔드, 모델 제공자, 기억 제공자, 프로토콜 인터페이스를 한 저장소에 모두 넣은 것은 Hermes의 매력이자 가장 큰 부채다. 작은 팀이라면 기능보다 조합 관리가 먼저 병목이 된다.

## Hermes가 특히 강한 지점

### 장기 세션 운영 감각

압축, 세션 계보, `FTS5`, 세션 검색, 기억 flush, 대시보드 분석을 보면 Hermes는 “긴 대화를 어떻게 운영할 것인가”를 실제 저장 구조와 제어 루프로 내려 보낸 저장소다.

### 표면 재사용 능력

CLI, 게이트웨이, `ACP`, API 서버, `MCP server`가 모두 공통 코어를 재사용한다는 점은 명백한 강점이다. 제품 범위가 넓어질수록 가치가 커진다.

### 운영자 중심 안전 모델

Hermes는 안전을 이유로 실행력을 크게 깎지 않는다. 대신 승인, 격리, 경로 검증, 프로필 분리, 토큰 보호, 로그 마스킹, 공급망 감사 같은 여러 장치를 조합한다. 고급 사용자 제품에는 이 철학이 잘 맞는다.

### 관측성과 운영 도구

회전 로그, 세션 태그, 상태 API, 대시보드, `hermes logs`는 많은 오픈소스 에이전트 저장소가 약한 부분이다. Hermes는 이 축을 꽤 일찍부터 제품 일부로 가져간 것으로 읽힌다.

## Hermes가 취약할 수 있는 지점

### 플랫폼 범위가 너무 넓다

메시징 플랫폼만 해도 매우 많고, 실행 백엔드와 제공자까지 합치면 조합 폭이 폭발한다. 이 정도 범위를 계속 지키려면 상당한 CI, 회귀 테스트, 유지보수 커뮤니티가 필요하다.

### 코어 변경 반경이 크다

중앙 파일이 크다는 것은 작은 변경도 여러 표면에 예상치 못한 영향을 줄 수 있다는 뜻이다. 이 문제는 기능이 늘수록 더 심해진다.

### 운영 부담이 사용자와 팀 모두에게 남는다

Hermes는 셀프호스팅과 자가 운영에 친화적이지만, 그만큼 설치, 토큰, 플랫폼 인증, 프로세스 관리, 원격 백엔드 설정 부담을 사용자와 팀이 떠안는다. 관리형 제품과는 다른 비용 구조다.

### 의존성·생태계 변화에 민감하다

모델 API, 메시징 SDK, 브라우저 스택, 원격 실행 서비스, `MCP` 서버 생태계는 모두 빠르게 변한다. 공급자 비종속 구조가 장점이면서도 유지보수 표면을 넓힌다.

## 차원별 `trade-off(절충점)`

### 안전성

Hermes의 다층 방어는 현실적이지만 복잡하다. 정책이 한 파일에 모이지 않고 여러 계층에 퍼져 있기 때문에, 보안 검토 자체는 더 어려워진다.

### `UX(사용자 경험)`

“어디서나 같은 에이전트를 쓴다”는 경험은 강하다. 대신 각 표면의 제약이 다르므로 완전히 동일한 UX를 기대할 수는 없다.

### 유지보수성

공통 코어 재사용은 장점이지만, 중심 파일이 커질수록 변경 비용이 급격히 늘어난다. 재사용과 응집의 균형이 시간이 갈수록 더 어려워진다.

### 이식성

Python 중심, 선택적 Node 보조 구조는 플랫폼 범위를 넓히는 데 유리하다. 하지만 WhatsApp 브리지, 브라우저 설치, 음성 처리, 특정 원격 백엔드는 결국 시스템별 예외를 만든다.

### 관측 가능성

세션 저장과 로그, 분석 대시보드가 있는 것은 장점이다. 다만 운영 이벤트가 여러 저장소에 흩어져 있어, 완전히 통합된 관측 플랫폼으로 보기에는 아직 거리가 있다.

## 어떤 조직에 특히 맞는가

Hermes는 다음 조건에서 특히 잘 맞는다.

- 개발자 중심 팀이 직접 운영하는 에이전트 제품
- 하나의 에이전트를 여러 채널과 워크스페이스에서 이어 써야 하는 경우
- 장기 세션, 자동화, 원격 실행이 핵심 가치인 경우
- 모델 제공자와 실행 백엔드가 자주 바뀔 가능성이 높은 경우

반대로 다음 조건에서는 약한 편이다.

- 폐쇄형 보안 정책과 강한 중앙 통제를 우선하는 환경
- 사용자가 인프라와 인증 설정을 거의 손대지 않길 원하는 제품
- 단일 웹 UI와 짧은 세션만 있으면 되는 사용 시나리오
- 작은 팀이 빠르게 단순 제품을 내야 하는 초기 단계

## 작은 팀이라면 무엇부터 버려야 하나

리소스가 제한된 팀이라면 아래 항목은 과감히 미루는 편이 좋다.

- 다수의 메시징 플랫폼
- 세 개 이상 실행 백엔드
- 복수 제공자 체인
- `MCP server` 표면
- 문맥 엔진 교체 기능
- 운영 대시보드의 풀세트 기능

대신 아래는 가능한 빨리 넣는 편이 좋다.

- 세션 저장과 검색
- 승인과 경로 보안
- 최소 `tool registry`
- 장기 세션 압축
- 인스턴스별 상태 분리
- 기본 로그와 상태 조회 경로

## 엔터프라이즈 팀과 소규모 팀이 다르게 볼 지점

소규모 팀은 Hermes의 폭보다 뼈대를 봐야 한다. 공통 코어, 세션 저장, 승인, 로그만 가져와도 충분히 많은 가치를 얻는다.

반대로 엔터프라이즈 팀은 다음 지점을 더 높게 볼 수 있다.

- 프로필 기반 격리
- 공급망 감사와 빌드 경로
- 다표면 접속 구조
- 운영 콘솔과 로그 체계
- 원격 실행 백엔드 분리

즉 같은 설계를 보더라도, 작은 팀은 단순화 관점에서, 큰 팀은 통제와 운영 관점에서 읽는 편이 맞다.

## 재구현 팀을 위한 체크리스트

- 공통 코어가 정말 하나여야 하는가, 아니면 두 개의 명시적 코어가 더 안전한가
- 세션 저장소를 로그가 아니라 검색 가능한 핵심 저장 체계로 볼 준비가 되어 있는가
- 모델 제공자 변화에 얼마나 노출될 것인가
- 실행 백엔드 수를 제품 가치와 직접 연결할 수 있는가
- 메시징 게이트웨이의 인터럽트·승인·큐잉을 운영할 수 있는가
- `skill`과 `tool`의 경계를 문서화할 수 있는가
- 프로필 또는 인스턴스 분리를 초기에 넣을 것인가
- 로그와 상태 API를 어느 시점에 붙일 것인가
- 공급망 감사와 의존성 검토까지 CI에 포함할 것인가

이 질문들에 답하지 못한 상태에서 Hermes류 시스템을 만들면, 대부분의 문제는 모델 품질이 아니라 운영 구조에서 터진다.

## 최종 합성

- `Top strengths(핵심 강점)`
  공통 코어 재사용, 장기 세션 저장·검색 구조, 운영자 중심 안전 장치, 실행 위치 분리, 운영 콘솔과 로그 체계가 가장 강하다.
- `Top limitations(핵심 한계)`
  거대한 중심 파일, 지나치게 넓은 조합 폭, 분산 상태 저장, 높은 운영 부담이 가장 큰 약점이다.
- `Most credible differentiators(가장 신뢰할 만한 차별점)`
  메시징·IDE·HTTP·`MCP`를 하나의 세션 코어로 묶는 구조, 세션 계보와 압축까지 포함한 장기 운영 감각, 원격 실행 백엔드 추상화가 가장 신뢰할 만한 차별점이다.
- `Best-fit adoption scenarios(가장 잘 맞는 도입 상황)`
  개발자 운영 중심 에이전트 허브, 장기 세션과 자동화가 중요한 개인 또는 소규모 팀, 여러 채널을 하나의 실행 주체로 묶어야 하는 제품에 가장 잘 맞는다.
- `Worst-fit or weak-fit scenarios(잘 맞지 않거나 제한적으로만 맞는 도입 상황)`
  완전 관리형 소비자 앱, 강한 중앙 보안 통제가 필요한 폐쇄 환경, 단순 웹 채팅만 있으면 되는 제품, 작은 팀의 초반 MVP에는 잘 맞지 않는다.
- `What to borrow first(먼저 가져갈 것)`
  `AIAgent`식 공통 코어 구조, `SessionDB` 같은 검색 가능한 세션 저장소, 승인·경로 보안, 최소 로그/상태 조회 체계를 먼저 가져가는 편이 좋다.
- `What to redesign first(먼저 다시 설계할 것)`
  초대형 중앙 파일 구조, 분산된 설정 해석, 지나치게 넓은 플랫폼/백엔드 범위는 그대로 복제하지 말고 초기에 더 좁고 명시적으로 재설계하는 편이 낫다.

## 요약

Hermes는 아이디어가 많은 저장소가 아니라, 이미 상당 부분 제품 문제를 떠안은 운영 중심 에이전트 런타임이다. 여기서 가장 배울 만한 것은 세션, 권한, 실행, 관측을 한 시스템으로 다루는 태도이고, 가장 조심해야 할 것은 그 태도를 구현하는 과정에서 코어가 지나치게 비대해졌다는 점이다. 비슷한 시스템을 새로 만들 팀이라면 Hermes를 완성본으로 복제하기보다, “세션과 실행 경계를 먼저 설계하고 기능 인터페이스는 천천히 넓힌다”는 원칙만큼은 반드시 가져가는 편이 낫다.
